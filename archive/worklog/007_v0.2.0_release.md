## 2. 무엇을 의도적으로 미진입했는가 (worklog/006 결정 그대로)

### 2-1. LibreOffice 경유 4포맷 (DOC/PPTX/PPT/이미지)

#### 표면 근거
- HWP/HWPX 가 60%+ 도메인 분포 (사용자 데이터)

#### 진짜 근거
- system 의존성(soffice CLI) 도입 시 사용자 `pip install` 경험이 두 종류로 갈라짐
- CI 매트릭스(Ubuntu LTS / Windows latest × Python 3.11) 변경 없이 v0.2 마감 가능

→ v0.3 으로 분리.

### 2-2. HWP COM 폴백 (Windows + 한/글 설치 환경)

#### 합격 기준
- pyhwp 단독 통과율 ≥ 90% → COM 폴백 미진입 (worklog/006 § 6-1)

#### 실측
- HWP 픽스처 **5/5 통과 = 100%**. 추출 결과 모두 도메인 키워드 매칭 + 표 47~90건 정확

→ COM 폴백 / kordoc 포팅 모두 v0.2 에서 미진입. 향후 새 도메인 진입 시 통과율
재측정 후 결정.

### 2-3. HWP 이미지 추출

XHTML 에 `bindata/` 경로로 이미지가 포함되지만 `ExtractedImage` 로 승격 안 함.
이유: HWP 의 페이지 좌표계와 `ExtractedImage.bbox` 의 PDF-style 좌표계가 비자명
매핑. `.hwp.images = []` 가 v0.2.0 의 contract. v0.3 의 후속 결정 사항.

### 2-4. PDF CCITTFax 픽스처

reportlab 으로 합성 어려움. 실 스캔 PDF 픽스처를 도입한 시점에 자연 진입 — 현재
회귀 차단 없음.

## 3. v0.1.1 의 "표면 근거 ≠ 진짜 근거" 사례가 또 발견됨

### 3-1. lets_portal 의 GPL v3 표기는 부정확이었음

`worklog/003` § 4 와 `packages/hwp/pyproject.toml` 모두 pyhwp 를 GPL v3 으로 적었으나,
**실제 라이선스는 AGPLv3+**. 1차 작업 시점에 정확히 확인 안 함.

#### 영향
- 사내 private 라이브러리이므로 실 영향 0
- 외부 공개 검토 시점에 AGPL viral 영향 (network use clause) 재검토 필요

#### 정정
- `packages/hwp/pyproject.toml` license / classifiers 갱신
- `CHANGELOG.md` [0.2.0] § Changed 에 명시

### 3-2. HWP 픽스처 "합성 불가능" 표면 근거 검증

`worklog/003` § 4 가 "합성 HWP 픽스처는 불가능" 이라고 했고, v0.2 에서 실제로
합성 시도 안 하고 lets_portal 의 9건 회수로 대응. 단, 시도 자체를 안 한 것은
`worklog/005` § 3 의 "30분 정도 실제 시도해본 뒤 결론" 원칙에 어긋남.

다만 HWP 5.0 바이너리 spec 의 복잡도 (OLE compound document + 한컴 자체 압축
포맷) 를 감안하면 30분 합성 시도가 의미 있을지 의문. 향후 시도 자체가 학습
가치를 주는 시점에 재시도 — 지금은 우선순위 아님.

## 4. 새 의존성의 진짜 비용

### 4-1. pypdf (core)

- BSD-3, ~2MB, OS 무관. CI 시간 영향 < 1초
- pdfplumber 와의 per-page 인덱스 매칭이 안정적 (테스트 픽스처 4건에서 검증)
- 단, count 불일치 시 방어 fallback 코드가 늘어남 (`pdf.py` 95% coverage 의 일부 미커버 분기)

### 4-2. pyhwp (extras)

- AGPLv3+, ~5MB
- **`six` 누락 packaging bug** — pyhwp 0.1b15 의 declared deps 에 six 가 빠져 있음.
  우리가 명시적으로 `six>=1.16` 추가해서 우회. pyhwp 가 업데이트되면 제거 검토
- `beautifulsoup4` + `lxml` transitive — XHTML 워크용. 이미 다른 Python 프로젝트에서
  널리 쓰이는 조합이라 친숙도 높음
- **HWP 5건 픽스처 회귀 시간 ~5분** — 실 HWP 파싱이 무거움 (~30~170초/건). 개발
  사이클이 5분 단위로 길어짐. v0.2.x patch 에서 session-scoped 캐싱으로 최적화 여지

## 5. 다음 세션이 알아야 할 것 (v0.3 진입 가이드)

### 5-1. v0.3 의 첫 의사결정 — LibreOffice 의존성 전략

worklog/006 § 3-3 의 표를 다시 펼치는 시점. 이번엔 system 의존성을 다룸:
- 후보 1: `soffice` CLI 호출 (subprocess)
- 후보 2: pyodide-libreoffice 같은 임베디드 (가능 여부 미확인)
- 후보 3: python-pptx / python-docx 직접 사용 (PPTX 만, DOC/PPT 는 불가)

CLAUDE.md §"경쟁 라이브러리 검토 의무" — 최소 3개 비교 필요.

### 5-2. HWP 회귀 시간 최적화

`packages/hwp/tests/conftest.py` 의 fixture 가 path 만 반환 — 각 테스트가 다시
`extract(path)` 호출. session-scoped 로 `extract()` 결과 자체를 캐싱하면 1건당 30~170초
대신 1초로 떨어짐. v0.2.x patch 후보.

### 5-3. v0.2.0 baseline 사용법

`benchmarks/results/v0.2.0_baseline.json` 가 박혔으므로, v0.2.x patch / v0.3 진입 시:

```bash
uv run python benchmarks/compare.py --fixtures samples --adapters korean_doc_parser
# 새 결과 vs v0.2.0_baseline.json 의 mean_duration_ms / mean_peak_mem_mb 가
# ±5% 안인지 확인 (CLAUDE.md §"라이브러리 특화 규칙 §6")
```

기준 수치 (v0.2.0):
- mean_duration_ms: **52,942**
- mean_peak_mem_mb: **44.16**
- composite: **0.910**

회귀 정당화 가능: 새 기능 추가 (이미지/표/메타) 시 처리량이 늘어나는 게 자연스러움.
±5% 룰을 기계적 적용 말 것 — 정당화는 PR 본문에 명시.

### 5-4. PPTX 파서가 PdfParser 어댑터에 들어 있지 않은 이유

v0.1.x 부터 core 가 `.pptx` 미지원. v0.3 의 LibreOffice 경유로 채울 예정.
`legacy_file_parser` 는 `.pptx` 를 LibreOffice + python-pptx 조합으로 처리하지만,
우리는 그 코드를 ports 안 한 상태. v0.3 의 의사결정 사항.
