## 4. 표면 근거 vs 진짜 근거

### 표면 근거
- HWP/HWPX 60%+ 도메인 분포 → HWP 우선이 자명.
- pypdf 가 가볍고(2MB) BSD-3 호환 → 채택 자연스러움.
- v0.1.1 가 새 의존성 0건으로 마감됐으므로 v0.2 에서 의존성 boundary 를 한 번에 정리하는 것이 정합적 (005 § 2 의 이월 근거와 일치).

### 진짜 근거
1. **boundary 의 종류 격리** — LibreOffice 를 v0.2 에 포함하면 의존성이 두 종류(Python 패키지 + system binary) 로 갈라짐. 사용자의 `pip install` 경험이 한 번 더 갈라지면 라이브러리 채택 비용이 비선형으로 증가. HWP 만 먼저 안정화하면 모든 v0.2 의존성이 **Python 패키지 한 종류**로 통일됨.

2. **CI 매트릭스 폭증 회피** — LibreOffice 통합 시 Ubuntu 의 soffice 버전 / Windows 의 LibreOffice 설치 경로 / 변환 결과의 OS-간 차이가 픽스처에 영향. v0.2 에 HWP+PDF만 들어가면 기존 매트릭스(Ubuntu LTS + Win latest × Python 3.11) 변경 없이 유지 가능.

3. **bidwatch 미정 시그널의 의미** — 외부 데드라인이 없으므로 "빨리 7포맷 채우기" 가 아니라 "boundary 를 신중하게 설계하기" 가 우선. 도메인 신호(HWP 60%+) 가 우선순위를 결정하고, 일정 압박 부재가 **scope 축소**를 정당화함.

4. **pyhwp 단독으로 충분한지 확인 완료** — `worklog/003` 에서 Linux 단독 동작 검증됨. COM 폴백을 v0.2 의 합격 기준에 넣으면 Windows-only 코드 경로가 늘어나서 OS 매트릭스 부담이 또 커짐. **합격 기준에서 COM 폴백 제외**, 픽스처 통과율이 떨어질 때만 옵션 추가.

## 6. 잠재 과제

### 6-1. pyhwp 한계 발견 시 분기점
- 픽스처 통과율 ≤ 70% → kordoc 포팅 진입 (`worklog/004` 의 차용 의무 절차 트리거)
- 70 < 통과율 < 90% → COM 폴백 옵션 추가 검토 (Windows 한정)
- 통과율 ≥ 90% → pyhwp 단독으로 v0.2 마감

### 6-2. PDF 비트맵의 색공간 / 필터 edge case
- pypdf 가 `/DCTDecode` (JPEG) / `/FlateDecode` (PNG-like) 외 case 에서 어떻게 처리하는지 사전 조사
- `/CCITTFaxDecode` (Fax/TIFF) 는 입찰공고 스캔 PDF 에 종종 나타남 — 픽스처 필수 1건
- 색공간 `/DeviceCMYK` 변환은 PIL 필요 (이미 dep 에 있음, 추가 0)

### 6-3. LibreOffice v0.3 의 사전 정지작업
- v0.2 작업 중 LibreOffice 픽스처를 위한 별도 합성 스크립트를 만들지 말 것 (yagni).
- 단, v0.1.0 CI 의 `integration` job (LibreOffice on Ubuntu, `-m slow`) 은 유지 — v0.3 진입 시 즉시 가동.
