## 2. SemVer 영향 분석

### 2-1. 호환 가능한 추가

- `korean_doc_parser.pipeline` 신규 sub-package — 기존 코드 무관
- `ExtractedImage.bbox_unit` 신규 필드 — frozen dataclass 의 필드 추가는
  ExtractedImage 직접 생성 코드에 영향 (default 없음). 그러나 라이브러리
  사용자가 ExtractedImage 를 직접 만드는 케이스는 거의 없음 (파서가 반환만)
- `VisionCache.stats()` 반환에 `total_hit_count` / `hit_rate` / per-model
  `hit_count` 추가 — dict 새 키 추가는 호환 가능
- `VisionCache` schema 의 `hit_count` 컬럼 — 마이그레이션 자동 (worklog/019 § 3-4)

### 2-2. 호환 가능한 동작 변경

- **default 모델 Sonnet → Haiku** — 호출자 코드 변경 0건, 비용 1/10.6 감소 +
  품질 동등 (worklog/014). 정밀 필요 시 `--model claude-sonnet-4-5` 명시.

### 2-3. 호환성 깨질 가능성 있는 항목 (low impact)

- **`ExtractedImage(bbox_unit=...)` 키워드 인자가 필수**. ExtractedImage 직접
  생성하는 외부 코드가 있다면 갱신 필요. **추정 영향: 0건** — bidwatch /
  vanasso.kr 가 라이브러리 반환값만 받아서 사용, 직접 생성 안 함
- `default` 모델이 Sonnet 가정한 호출자 코드 — 모두 명시적 `--model` 또는
  `VisionClient(model=...)` 호출하므로 영향 0

## 3. 추가된 public API (`korean_doc_parser.pipeline`)

```python
from korean_doc_parser.pipeline import (
    compute_doc_id,            # path → sha256 64-char hex
    weighted_confidence,       # 3-channel blend, default (0.4, 0.2, 0.4)
    DEFAULT_WEIGHTS,           # 위 default 값 노출

    detect_caption_regex,      # text → (caption, score) 한국어 + 영문 패턴
    detect_caption_proximity,  # bbox + text_blocks → (caption, score)
    TextBlock,                 # proximity 입력용 dataclass
    CaptionDetection,          # (예약) 미래 detector 의 결과 형식
)
```

**호출자 사용 패턴 (bidwatch):**

```python
from korean_doc_parser import extract
from korean_doc_parser.pipeline import (
    compute_doc_id,
    weighted_confidence,
    detect_caption_regex,
)
from korean_doc_parser.vision import VisionClient

result = extract(notice_file)
doc_id = compute_doc_id(notice_file)

for img in result.images:
    # 1차 검출 (parser 가 이미 채웠을 수도 있고, 호출자가 더 풍부한 text 로
    # 다시 호출할 수도)
    cap, regex_score = detect_caption_regex(img.text_before + "\n" + img.text_after)

    # Vision (default = Haiku, 비용 1/10)
    vr = VisionClient().label(img.file_path)

    # 3중 가중
    final = weighted_confidence(
        regex_score=regex_score,
        proximity_score=img.caption_pattern_score,  # parser 가 채웠다면
        vision_confidence=vr.confidence,
    )
    bidwatch_db.images.insert(
        doc_id=doc_id, sha256=vr.sha256, caption=vr.caption or cap,
        confidence=final, auto_approved=final >= 0.7, ...
    )
```

→ **호출자가 자체 DB 책임. 라이브러리는 알고리즘만.**

## 4. 사용자 입장의 마이그레이션 가이드

### 4-1. v0.4.5 → v0.5.0 직접 영향 — 없음 (예상)

대부분 호출 패턴은 그대로:
- `extract(path)` — 동일
- `VisionClient(...).label(...)` — 동일 (default 모델만 Haiku 로 바뀜)
- `kdp-label` CLI — 동일 (default 모델 Haiku)

### 4-2. 사용자가 명시적으로 Sonnet 을 원하면

```bash
kdp-label image.png --model claude-sonnet-4-5
```

또는 Python:
```python
VisionClient(model="claude-sonnet-4-5", cache=...)
```

worklog/014 의 22건 실측에서 한국 차트 caption 품질은 동등 (93%/93%), Haiku 가
오히려 calibration 정직 → **별도 이유 없으면 default 유지 권고**.

### 4-3. ExtractedImage 직접 생성 (라이브러리 내부 / 테스트 등)

새 필드 `bbox_unit` 필수. 가이드:
- bbox 가 픽셀이면 `bbox_unit="px"`
- bbox 가 EMU 면 `bbox_unit="emu"`
- bbox 가 None 이면 `bbox_unit="none"`

### 4-4. cache 의 hit_count 마이그레이션

기존 v0.4.x cache.db 가 있어도 자동 마이그레이션 (worklog/019 § 3-4 의
`_MIGRATIONS` 의 idempotent ALTER TABLE). 기존 row 들은 `hit_count = 0` 으로
시작 → 이후 호출부터 누적.

## 5. 잠재 과제 / 다음 세션이 알아야 할 것

### 5-1. 각 파서의 caption 1차 자동 통합 (v0.6+)

worklog/019 § 3-2 의 contract 는 "각 파서가 ExtractedImage 채울 때 caption 모듈
호출". v0.5.0 시점에는 **알고리즘 모듈만 박힘**, 각 파서의 자동 통합은 미진입.
호출자가 직접 `detect_caption_regex(img.text_before)` 호출하면 동등 효과 —
이 권고는 README/recipes 에 박을 것 (Phase 8).

### 5-2. proximity score 의 단위 호환 검증

`detect_caption_proximity()` 가 image_bbox + text_blocks 가 같은 단위라 가정.
호출자가 `bbox_unit` 을 확인 후 같은 단위만 묶어서 전달해야 함. 자동 검증
미진입 — 호출자 실수 가능. **README/recipes 에 박을 사항**.

### 5-3. cache hit_rate 의 해석

`hit_rate = total_hits / (total_hits + total_rows)`. 즉:
- 신규 row 가 적고 hit 이 많으면 hit_rate 가 1 에 가까움 (좋은 운영)
- 매번 새 이미지 들어오면 row 가 늘어 hit_rate 가 낮아짐 (정상)

→ **operator 가 추세를 봐야** 의미 있음. 절대값보다는 시간별 추이 (`by_date`
의 hit_count 추가는 v0.6+ 후보).

### 5-4. benchmarks/adapters/ 의 PPTX 미반영 (worklog/018 § 4-1)

v0.4.5 측정에서 PPTX 4 fixture skip. v0.5.0 release 후 Phase 8 (호출자 문서)
시점에 adapter 갱신 + 재측정 권고.
