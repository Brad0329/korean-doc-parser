## 2. SemVer 1.0.0 의 의미

CLAUDE.md §"라이브러리 특화 규칙 §1":
- **patch:** 버그 수정만
- **minor:** 호환 가능한 추가
- **major:** 호환 깨짐

v1.0.0 은 일반적인 major bump (호환 깨짐) 가 아닙니다. **호환성 보장의
시작점** 이라는 의미가 강함:

| v0.x.x 시절 | v1.0.0 이후 |
|---|---|
| 사내 internal milestone, 호환성 명시 없음 | **public API 의 호환성 보장 시작** |
| 매 patch 마다 ParseResult 필드 추가 가능 | breaking 변경은 v2.0.0 까지 미루기 (또는 deprecation cycle) |
| 의사결정 자유 | 호출자 (bidwatch / vanasso.kr / 사내 RAG) 의존성 우선 |

→ **이후 변경은 의식적인 trade-off** — "비용 절감 vs 호출자 영향" 명시.

## 5. v1.0.0 이후 운영 정책 (CLAUDE.md 부합)

### 5-1. 유지보수 흐름

CLAUDE.md §7 (삭제됨, "공개 OSS 가정 SLA") 가 아니라 **사내 운영 흐름**:

- bidwatch 운영 중 발견된 파싱 실패 → 회귀 테스트로 영구 추가
- 의존성 보안 알림 분기 1회 점검 (예: pyhwp / pdfplumber CVE)
- Python 새 버전은 사내 운영 버전 따라가기 (현 3.11 → 다음 3.12)
- 회귀 측정 (벤치마크 `compare.py`) — patch 마다 또는 분기 1회

### 5-2. 잠재 patch 후보 (호출자 가치 발생 시)

[`docs/internal/known-limitations.md`](../docs/internal/known-limitations.md)
의 5 카테고리 (A 검증, B v1.0 이후 patch, C 알고리즘 정밀화, D 측정 정밀화,
E 도구 자동화) 모두 살아있음. 각 항목은 bidwatch / 사내 RAG 운영 발견 시
진입 결정.

### 5-3. v2.0.0 이 필요해지는 시나리오

- `ExtractedImage` 필드 호환성 깨는 변경 (예: `bbox` 단위 통일 정규화)
- public API 의 함수 시그니처 변경
- pipeline 의 `weighted_confidence` 의 weights 기본값 변경 (호출자 동작 영향)
- → 모두 deprecation cycle (worklog 명시 + v1.x 의 patch 로 warning) 후
  v2.0.0 에서 실행
