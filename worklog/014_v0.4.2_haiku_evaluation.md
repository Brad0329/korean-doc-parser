## 3. 표면 결과 vs 진짜 분석

### 3-1. "한국 차트 caption 품질 우려" (worklog/011 A.1) 의 결말

| 표면 | 진짜 |
|---|---|

### 3-2. Sonnet 의 confidence calibration 평탄화 — 체계적 재확인

| 표면 | 진짜 |
|---|---|
| Sonnet 22/22 중 21건이 confidence ≥ 0.92 → "Sonnet 이 자신 있음" | **Sonnet 은 confidence calibration 이 평탄.** 빈 회색 이미지 (959145dd) 까지 conf 0.95 로 자신만만. 임계값 0.7 trigger 가 22건 중 0건 → **임계값의 의미가 Sonnet 에서는 죽음** |
| Haiku 가 13.6% 를 < 0.7 로 표시 → "Haiku 가 신뢰성 낮음" | **Haiku 가 정직하게 calibrate.** placeholder 같은 ambiguous 이미지에서 conf 0.3 으로 낮춤. 2-pass reasoning 호출이 처음으로 동작 (worklog/011 B.1 의 2-pass 전략 검증) |

**v0.5 함의 (검수 큐 정렬):** worklog/011 § E "신뢰도 낮은 순" 정렬 기준이 의미를
가지려면 **Haiku 가 default 여야 함**. Sonnet 에서는 모든 이미지가 0.92+ 라
정렬이 noise. v0.5.0 default 전환의 추가 근거.

### 3-3. image_type 불일치 84.2% 의 의미

| 표면 | 진짜 |
|---|---|
| 16/19 = 84.2% agreement = "Haiku 의 정확도가 약간 떨어짐" | **모호 케이스가 절반.** 3건 중 1건 Sonnet 우세, 1건 Haiku 우세, 1건 무승부. 양 모델 모두 photo/screenshot/diagram 의 경계 케이스에서 자기 선호로 분류. 정답이 없는 카테고리 정의 문제 |

→ image_type 의 5-카테고리 (chart/photo/diagram/screenshot/logo + other) 가
실 이미지에서 **상호 배타적이지 않음**. v0.5 의 검수 큐가 이 ambiguity 를
수렴시킬 메커니즘.

### 3-4. 비용 1/10.6 의 정확한 수치

| 표면 | 진짜 |
|---|---|
| "Haiku 는 1/10 비용" (pricing.py 의 토큰 단가 비율) | **실측 1/10.6** — 토큰 단가만 보면 Haiku input $0.25/M vs Sonnet $3.0/M = 1/12. 그러나 같은 이미지에 대해 **Haiku 의 output 토큰이 약간 더 길음** (caption 풀어 씀) → 토큰 비례 보정 후 1/10.6 |

→ 비용 모델은 token 단가가 아니라 **실 호출당 비용 측정**이 정확. v0.5 의 cost_log
구축 시 token-level + call-level 양쪽 저장 필요.

## 5. v0.4.x 잠재 과제 / 다음 세션이 알아야 할 것

### 5-1. PPTX 이미지 추출 0건 — markitdown 위임 한계

samples/ 의 PPTX 4종 모두 `extract().images` 가 0건:
- `한국관광공사_제안서.pptx`
- `pptx_qvan_storyboard.pptx`
- `pptx_letsportal_tmp.pptx`
- `pptx_vanasso_upload1.pptx` (BadZipFile — 손상)

**원인 가설:** v0.3.0 의 PPTX 파서가 markitdown 위임 (worklog/010) → markitdown 이
이미지를 텍스트 placeholder 로만 변환, 비트맵 추출 없음.

**영향:** v0.4 Vision CLI 의 `--from-document` 가 PPTX 에서 작동 안 함.
**v0.5 또는 v0.4.3 잠재 과제:** PPTX 비트맵 추출 직접 구현 (python-pptx 의
shape.image API) 또는 markitdown patch.

### 5-2. PDF 중복 이미지 정책 — Pipeline 결정 항목

키움증권 PDF 에서 동일 sha256 이 4페이지에 등장. 현재 extract 는 모두 반환 →
22 이미지 중 unique 19. **Pipeline (v0.5) 에서 dedup 정책 결정 필요:**
- (a) extract 단계 dedup (메타데이터에 page_no 리스트 보존)
- (b) Pipeline 단계 dedup (sha256 캐시가 자동 처리)
- (b) 가 worklog/011 C.4 의 cache 정책과 자연 정합 → default 안

### 5-3. confidence 임계값 0.7 의 미래

22건 실 측정에서:
- Sonnet: 100% ≥ 0.7 → 임계값 무용
- Haiku: 86.4% ≥ 0.7 → 임계값 작동

**v0.5 default Haiku 전환 후:**
- 임계값 0.7 은 그대로 유지 (현재 13.6% trigger 가 합리적)
- 단, 검수 큐가 0.3-0.69 구간을 충분히 받는지 모니터링 필요
- 향후 임계값 동적 조정 (운영 데이터 누적 후) 가능성

### 5-4. Sonnet calibration 부족의 대처

Sonnet 을 명시적으로 선택한 사용자 (정밀 케이스) 에게는 confidence 가 의미 없음:
- option (a): Sonnet 호출 시 client 에 calibration 보정 multiplier 적용 (e.g. 0.85x)
- option (b): Sonnet 호출 시 항상 2-pass 강제 (reasoning 받아서 별도 판단)
- option (c): v0.5 검수 큐는 default Haiku 기준으로 설계, Sonnet 은 보조

**default 안: (c)** — 별도 코드 추가 없이 Haiku 를 1급 시민으로 다룸.

### 5-5. 평가 자동화 스크립트 보관 (scripts/eval_haiku.py)

이번 평가의 자동화 스크립트는 다음 모델 추가 시에도 재사용 가능:
- 향후 claude-opus-4-x 추가 시 → `--models claude-opus-4-x claude-haiku-4-5` 로 호출
- v0.5 회귀 비교 → 같은 22 이미지로 default 변경 전후 비교
- 픽스처 의존 0 — 인자로 받는 모든 PDF/PPTX/HWPX 에 적용 가능

산출물 (eval_v0.4.2/) 자체는 `.gitignore` (raw data 보존 안 함). 수치는 본
worklog 에만 박음.
