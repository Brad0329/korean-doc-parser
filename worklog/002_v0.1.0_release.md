## 2. 왜 7포맷 → 3포맷으로 축소했는가

**원래 v0.1.0 합격 기준** (HANDOVER §6):
> 7포맷(HWP/HWPX/PDF/DOCX/DOC/PPTX/PPT) 모두 텍스트 + 표 + 이미지 + 메타데이터 추출

**실제 출시:** 3포맷만.

### 표면 근거 (운영/시간)

- 남은 4포맷(HWP / DOC / PPT / PPTX)이 모두 **외부 환경 의존성**:
  - HWP: pyhwp (Python 3.6 까지만 공식 검증) + pywin32 + 한/글 설치
  - DOC / PPT / PPTX: LibreOffice binary + PATH 설정
- 환경 가변성이 높아 CI 매트릭스에서 안정 검증이 어려움 (특히 한/글 COM 폴백)
- 핵심 영역 3포맷이 이미 라이브러리의 "RAG-친화 마크다운 변환" 가치를 충분히 입증

### 진짜 근거 (아키텍처/의존성)

- **단일 파일 포터빌리티 + 패키지 경계** 원칙이 잘 동작하는지를 v0.1.0 으로 먼저 검증
- 4포맷 추가 시 의존성 매트릭스 (uv.lock 크기, CI 시간) 가 급격히 늘어남 → v0.1 에서는 가벼운 의존성만
- HWP/HWPX 차별화 가치는 **HWPX** 만으로도 90% 입증 (HWPX 는 현재 한/글 신규 포맷; HWP 는 레거시)
- LibreOffice 변환 경로는 markitdown 위임이 이미 Phase 0 에서 검증됨 → 실구현은 단순 wrapping 이지만, 환경 구성 검증이 별도 일거리

### 1차 작업의 PPTX 흔들림 교훈 반영

- 2026-04-22: PPTX 미지원 → 2026-05-14: 자동 변환으로 뒤집힘 (표면 근거 약화)
- 이번 결정의 **표면 근거 = 환경 의존성 부담**, **진짜 근거 = 의존성 매트릭스 가벼움**.
  표면이 흔들려도 진짜 근거는 유지됨 → 향후 v0.2 에서 추가해도 정합.

## 3. v0.2 회복 계획

**이월된 4포맷:**

| 포맷 | 패키지 | 의존성 | 비고 |
|---|---|---|---|
| HWP | `packages/hwp/` | pyhwp + pywin32 (Windows COM) | pyhwp 3.11 호환성 먼저 검증 필요 |
| DOC | `packages/core/` | LibreOffice → DOCX → 기존 파서 | _libreoffice.py 공용 컨버터 |
| PPT | `packages/core/` | LibreOffice → PPTX → markitdown | 같은 컨버터 재사용 |
| PPTX | `packages/core/` | markitdown[pptx] 위임 | LibreOffice 필요 없음 (이미 Office Open XML) |

**v0.2 합격 기준:**
- 4포맷 픽스처 각 최소 5건 통과
- LibreOffice 설치 자동화 (CI integration job 활성화)
- v0.1.0 의 3포맷 회귀 0건 (벤치마크 ±5% 이내)

**예상 작업량:** 1-2주 (HWP 환경 가변성 때문에 마진 크게).

## 4. 이번 마일스톤의 의사결정 / 실패 / 잠재 과제

### 결정 — 이미지 비트맵 추출 정책

- PDF: 비트맵 미추출 (file_path/sha256 placeholder, v0.2 이월). 이유: pdfplumber 가 비트맵 미제공 → pdf2image 또는 pypdf 추가 의존성 필요
- DOCX: 비트맵 추출 완료 (ZIP word/media 직접) — 단일 호출당 tempfile 누적. cleanup 책임 caller
- HWPX: 비트맵 추출 완료 (ZIP BinData/Resources 직접) — DOCX 와 같은 패턴

→ v0.2 에서 일관성 회복: PDF 도 비트맵 추출 + 모든 파서가 같은 tempfile cleanup 정책 (caller-provided output_dir).

### 결정 — table 데이터의 표현

ParsedTable 은 별도 객체로 두고 `result.markdown` 에 포함시키지 않음.

- 이유: 사용자가 원하는 직렬화 형식 (CSV/markdown/HTML) 이 다양. 라이브러리가 한 가지 선택 강제 안 함.
- 결과: GT 의 `expected_keywords` 가 표 내용을 가리키면 검증 실패 → GT 작성 시 명시 (paragraph 본문만 표적).
- v0.2 검토: 옵션 인자 `extract(path, inline_tables=True)` 로 markdown 안에 표 포함 모드 추가 여부.

### 실패 — initial PAT scope

GitHub PAT 의 `workflow` scope 누락으로 첫 CI yml push 가 거부됨.

- 원인: Git Credential Manager 가 자동 발급한 토큰의 scope 가 좁음
- 해결: 사용자가 Classic PAT 새로 만들고 `workflow` scope 부여 → Credential Manager 에 등록
- 교훈: 새 repo 첫 setup 시 PAT scope 명시적 확인 (특히 `.github/workflows/` 변경 예정이면 처음부터 workflow 포함)

### 잠재 과제 — coverage 85% 게이트 활성화

현재 `[tool.coverage.report]` 의 `fail_under` 가 주석 처리됨. v0.1.0 까지는 가시성만, v0.2 부터 활성화 검토.
HWPX 의 83% 가 85% 게이트에 살짝 모자라 — 이미지 픽스처 추가로 95%+ 가능.
