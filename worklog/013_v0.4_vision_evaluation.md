## 3. 표면 결과 vs 진짜 분석

### 표면 결과
- 5/5 합격 기준 통과 → v0.5 진입

### 진짜 분석

#### 3-1. 02_photo_like 오답의 진짜 원인

| 표면 | 진짜 |
|---|---|
| 모델이 photo 를 diagram 으로 오분류 | **합성 이미지가 photo 가 아님** — 그라데이션 + 도형 조합은 "그래픽 디자인" 에 가까움. Claude 의 분류가 더 정확 |

→ image_type 정확도 4/5 는 **모델의 한계가 아닌 픽스처 정의의 한계**. 실 사진을 테스트하면 더 높을 가능성.

#### 3-2. confidence 분포의 다양성 부족

- 모두 0.92-0.95 — auto_approved 임계값 0.7 의 효과 검증 불가
- 합성 이미지가 단순해서 Claude 가 자신 있게 분류
- **실 이미지 (스캔 PDF / 흐릿한 차트 / 한자 혼용 등) 로 confidence 분포 측정 필요** — v0.4.x patch 의 권고

#### 3-3. caption 의 부수 발견 — 한자 오인식

03_diagram 의 caption 에 "각 박스 중앙에 한자 '田' 이 표시되어 있습니다" — **우리는 한자 안 그렸음**. 박스 안 텍스트 ("입력", "처리", "출력") 가 한자 ‘田’ 으로 보였을 수도. malgun.ttf 한국어 폰트 렌더링이 작은 사이즈에서 깨졌을 가능성.

→ 폰트 / 텍스트 크기가 작으면 OCR-like 오해 발생. v0.5+ 의 실 OCR 도입 검토 시 참고.

#### 3-4. PII 마스킹 동작 검증

04_screenshot 의 caption: "사용자명 'admin'과 비밀번호가 마스킹된" — admin 은 PII 가 아니므로 우리 regex (주민번호/전화/이메일) 가 안 잡음. 정상 동작.

만약 caption 에 010-1234-5678 같은 패턴이 들어왔다면 010-XXXX-XXXX 로 마스킹됐을 것 (단위 테스트 `test_pii_in_caption_is_masked` 가 검증).

## 4. 합격 기준 외 부수 측정

### 4-1. Wall time
- 평균 4초/이미지 — end user 응답 시간 합리적 (단일 이미지)
- 100 이미지 문서 = 400초 = 6분 40초 → **비동기 처리 필수** (v0.5 D.2 결정과 정합)

### 4-2. Token 사용량
- 합성 이미지가 200x200 정도라 input ~374 tokens (저해상도 비용 영역)
- output 평균 ~100 tokens (caption + image_type + confidence 만)
- 실 이미지 (1000x1000+) 는 input ~1500 tokens 예상 → 비용 3-4배 증가 가능
- **실 이미지 비용 추정**: 3.75원 × 4 = **15원/이미지** (여전히 합격선 13.8원 근처)

### 4-3. 캐시 미적용 비용 vs 적용 시 절감

5건 같은 이미지 두 번째 호출 시 → 0원 (캐시 hit). 만약 같은 문서를 다시 처리하면 전체 비용 0.

### 6-2. 임계값 0.7 의 의미 재검토

평가에서 모든 이미지가 confidence ≥ 0.92 — 0.7 임계값은 trigger 안 됨. 가능성:
- (a) Sonnet 이 모든 케이스에서 confidence 를 0.9+ 로 부풀림 (calibration 문제)
- (b) 우리 픽스처가 너무 단순
- (c) 임계값 0.7 이 너무 낮음 — 0.85 / 0.9 로 올려야 의미

실 이미지 평가 후 다시 결정.

### 6-3. cli/label.py coverage 56% 의 의미

mock 테스트 (174 통과) 가 `_run_from_document` 미커버. 평가 단계에서 자연 커버됨 (실 PDF → extract → Vision 호출). v0.5 진입 전 e2e 테스트 추가 권고.
