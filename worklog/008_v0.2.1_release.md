## 2. 최적화 방법 — fixture 2단 분리

### 2-1. 무엇이 느렸나

v0.2.0 의 `packages/hwp/tests/conftest.py` 는 fixture 가 `Path` 만 반환:

```python
@pytest.fixture(scope="session")
def hwp_test_nara() -> Path:
    return _hwp_or_skip("test_nara.hwp")
```

각 테스트가 `extract(path)` 를 직접 호출 → 같은 HWP 가 여러 번 파싱됨.
프로파일 결과:

| HWP 파일 | 1회 extract 시간 | 호출 횟수 (v0.2.0) | 합계 |
|---|---|---|---|
| `test_nara.hwp` | ~50s | **3** (matches_GT + Korean text + table columns) | 150s |
| `proposal_consulting` (3. 정성적) | ~75s | 1 | 75s |
| `forest_startup` | ~113s | 1 | 113s |
| `proposal_wku` | ~51s | 1 | 51s |
| `gyeongnam_fishery` | ~28s | 1 | 28s |
| **합계** | — | **7회 extract** | **~417s** |

`test_nara` 가 가장 작은 파일인데도 3번 호출되면서 가장 큰 낭비.

### 2-2. 무엇을 했나

각 HWP fixture 를 path/result 2개로 분리:

```python
# Path (cheap, registration / corrupt 테스트용)
@pytest.fixture(scope="session")
def hwp_test_nara() -> Path: ...

# ParseResult (session-scoped, 1회 extract, 결과 공유)
@pytest.fixture(scope="session")
def hwp_test_nara_result(hwp_test_nara: Path) -> ParseResult:
    return _extract(hwp_test_nara)
```

테스트는 더 이상 `extract(path)` 를 호출하지 않고 `hwp_<name>_result` 의 `ParseResult`
를 직접 받음. session 내 1회만 파싱.

### 2-3. 표면 근거 vs 진짜 근거

#### 표면 근거
- worklog/007 § 5-2 가 명시한 후속 과제. 측정 결과 가시화된 통증.

#### 진짜 근거
- **회귀 시간이 5분 → 30초** 가 목표였으나 실제는 5분 53초 → 1분 59초 (66%).
  완전한 30초까지는 안 갔지만 patch 가치가 충분
- **개발 사이클의 marginal cost** 가 큰 폭으로 줄어듦 — TDD 친숙성 향상.
  v0.3 (LibreOffice) 작업 시 회귀 가드를 더 자주 돌릴 수 있음
- session-scoped 캐시는 **test 격리 위반 위험** 이 있음 (테스트가 ParseResult 를
  mutate 하면 다른 테스트에 영향). `ParseResult` 가 `frozen=True, slots=True`
  dataclass 라 mutate 안 됨 → 안전. 격리 위반 risk 0

## 3. 단축이 30초까지 안 간 이유 — 분석

5건 × 평균 1회 = 5번 extract. 평균 50초 × 5 = 250초 = 4분 10초 가 단순 예상.
실측은 120초 (2분) — 더 빠름.

추정 원인:
- **OS 파일 캐시** — 두 번째 측정이라 파일 시스템 cache 가 따뜻함
- **Python module cache** — pyhwp import 가 한 번만 일어남
- **단일 측정 변동성** — 다회 측정 시 평균은 더 높을 수 있음 (±20% 변동 예상)

다음 patch (v0.2.2 등) 에서 진짜 30초까지 갈 옵션:
1. `pytest-xdist` 로 5건 병렬 — 가장 큰 효과 예상 (수 분 → 가장 큰 1건 시간만 = 113초)
2. `pyhwp` 의 `HTMLTransform` 의 styles.css 생성 비활성화 — 우리는 안 쓰는데 생성됨
3. HWP 픽스처 N 축소 — 회귀 가드 가치 vs 시간 트레이드오프

지금 v0.2.1 에서는 1번 (병렬) 은 cross-platform 이슈 (Windows multiprocessing
시 fixture 직렬화 등) 가 복잡해서 보류. 진짜 30초 목표는 v0.3 작업 전에 재검토.

## 4. 잠재 과제

### 4-1. CI 시간은 변화 없음

samples/ 가 git untracked → CI 에서는 HWP 픽스처가 `pytest.skip`. CI 회귀 시간은
v0.2.0 과 v0.2.1 모두 동일 (수 초). 이번 최적화는 **로컬 개발 사이클** 전용.

### 4-2. 다른 패키지(core 등) 도 같은 패턴 적용 가능

PDF / DOCX / HWPX 의 fixture 는 합성이라 빠름 (각 < 0.1초). path-only fixture
유지로 충분. 회귀 가치 없음.

### 4-3. v0.2.0 baseline 의 mean_duration_ms 는 변동 없음

벤치마크는 `extract()` 1회 호출의 wall time 만 측정 — fixture 캐싱과 무관.
`v0.2.0_baseline.json` 의 mean 52.9s 는 v0.2.1 에서도 유효.
