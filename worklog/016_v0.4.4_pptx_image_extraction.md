## 2. 설계 결정

### 2-1. text 는 markitdown, bitmap 은 python-pptx — 둘 다 사용

| 표면 | 진짜 |
|---|---|
| markitdown 으로 bitmap 도 추출할 수 있을 듯 (slide-by-slide convert) | markitdown 은 ``![](filename)`` text placeholder 만 emit, blob bytes 미반환. 코드 변경 없이 그대로 두면 영원히 이미지 0건 (worklog/014 § 5-1) → python-pptx 직접 접근 필요 |

python-pptx 는 `markitdown[pptx]` extras 의 sub-dependency 라 새 의존성 0건.
`pip install korean-doc-parser[pptx]` 만으로 이미 PPTX 사용자에게 사용 가능.

### 2-2. shape walk + group 재귀

PPTX 의 shape 트리는 group 으로 중첩 가능. `_iter_picture_shapes()` 는
`MSO_SHAPE_TYPE.PICTURE` 만 yield 하되 `MSO_SHAPE_TYPE.GROUP` 은 재귀 walk.
다른 shape (text, autoshape, table, chart, smartart) 은 무시.

**v0.5+ 고려:** chart / smartart 는 vector 비트맵으로 변환 가능 (slide thumbnail
방식). 현재는 PICTURE shape 만 — 사용자가 끼워 넣은 명시적 비트맵이 1차 가치.

### 2-3. notes / master / layout 슬라이드는 제외

`prs.slides` 만 iterate. notes (발표자 노트), master / layout (배경 로고/푸터)
는 비즈니스 가치 적음 + 운영 시 중복 라벨링 비용 발생. v0.5 검수 큐가 이 정책을
재검토할 가능성 있음.

### 2-4. bbox 단위 = EMU (English Metric Unit)

PPTX shape 의 `left/top/width/height` 가 native EMU (1 inch = 914400 EMU).
PDF 파서는 px 단위라 **두 파서의 bbox 단위가 다름**. 통일 정규화는 v0.5+ 의
Pipeline 측 결정 항목 (worklog/014 § 5-2 의 dedup 결정과 같이 처리).

이 시점에 의도적으로 정규화 안 한 이유: 슬라이드의 viewport 픽셀 변환은 DPI
정의 필요 (96 DPI 가 표준이지만 PPTX 의 slide size 마다 다를 수 있음). EMU
는 lossless, 단위 메타만 별도 노출 (v0.5+ 의 `bbox_unit` 필드 후보).

### 2-5. 비표준 포맷 (.wdp / HD Photo) graceful degrade

`ppt/media/` 에 `hdphoto*.wdp` 가 끼어있을 때가 있음 (Microsoft JPEG XR).
PIL native 지원 안 함 → `_safe_image_size()` 가 `(0, 0)` 반환. **`sha256` +
`file_path` 는 보존** → 다운스트림 Vision 은 그대로 호출 가능 (Anthropic API 는
mime_type 만 맞으면 통과). 단, 실측에서 samples 3종 모두 `width=0` 케이스 0건
— shape API 가 .wdp 를 자동으로 PNG fallback 으로 변환해 반환하는 것으로 보임.

### 2-6. SemVer 판단 — patch (0.4.3 → 0.4.4)

엄격하게는 `ParseResult.images` 의 PPTX 동작 변경이 "호환 가능한 추가" →
minor. 그러나:
- v0.5.0 은 Pipeline + default Haiku 로 이미 박힘 (worklog/014 § 4-2)
- 자산화 경로의 patch 흐름 (v0.4.2 / v0.4.3 / v0.4.4) 일관성
- `images == []` 가정한 사용자 코드는 없을 것 (정상 사용에서 더 채워지는 건 환영)

→ **patch (v0.4.4)** 로 결정. CHANGELOG 에 동작 변경 명시. v0.4.2 의 default
전환 v0.5.0 이월 결정과 같은 패턴 (호환 가능한 변경은 minor 까지 미루기).

## 3. 무엇이 채워지는가 — `ExtractedImage` 필드

| 필드 | 값 |
|---|---|
| `page_no` | 1-based 슬라이드 인덱스 |
| `section_no` | None |
| `bbox` | (left, top, right, bottom) in EMU. None if any coord missing |
| `order_in_page` | 슬라이드 안 picture 순서 (1-based, group walk 후 flat) |
| `text_before` / `text_after` / `section_title` | "" / "" / None (PPTX 의 어느 텍스트박스가 caption 인지 결정 어려움 — 검수 큐 단계 책임) |
| `file_path` | tempfile (`kdp_pptx_s{slide}_{order}_*.{ext}`) |
| `sha256` | 64-char hex, blob 의 sha256 |
| `width / height` | PIL Image 사이즈. 비표준 포맷이면 (0, 0) |
| `size_bytes` | `len(blob)` |
| `mime_type` | `shape.image.content_type` 또는 `application/octet-stream` |
| `detected_caption` / `caption_method` / `caption_pattern_score` | None / None / 0.0 |

## 4. 잠재 과제 / 다음 세션이 알아야 할 것

### 4-1. 같은 이미지의 슬라이드별 중복

samples/qvan_storyboard 의 499 이미지 = 50 unique sha256 × 평균 10 슬라이드.
Vision 라벨링 비용을 최소화하려면 sha256 dedup 필요. **두 단계 dedup 정책:**

| 위치 | 효과 |
|---|---|
| Engine (parsers/pptx.py) | extract 단계에서 unique sha256 만 반환 |
| Pipeline (v0.5) | sha256 캐시가 자동 처리 — 이미 worklog/011 C.4 박힘 |

→ **engine 단계는 raw, dedup 은 Pipeline 책임**. worklog/014 § 5-2 와 동일
결정. 단, kdp-label CLI 만 쓰는 사용자는 같은 이미지 N회 라벨 (캐시 hit 으로
2회차 이후 0원이지만 row 늘어남).

### 4-2. PPTX 의 chart / smartart 추출

현재 PICTURE shape 만. chart / smartart 는 별도 XML (chart1.xml) 로 저장돼
있어 PNG 로 직접 렌더링 안 됨. **v0.5+ 후보**:
- LibreOffice 변환? — **CLAUDE.md §"금지 의존성" 위반** → 금지
- python-pptx 의 chart API + matplotlib 으로 자체 렌더링? — 복잡, 효용 작음
- 슬라이드 전체를 PNG 로 렌더링? — viewport DPI 결정 + python-pptx 한계
- → 일단 미진입. bidwatch 가 실제로 필요해지면 그때 결정

### 4-3. bbox 단위 통일 — v0.5 Pipeline 결정 항목

| 파서 | bbox 단위 |
|---|---|
| PDF | px (pdfplumber native) |
| HWPX | None (현재 박지 않음) |
| HWP | None |
| DOCX | None |
| PPTX | **EMU** (v0.4.4 신규) |

v0.5 의 Pipeline 이 정규화 정책 결정 (px / 비율 / 절대 좌표 / 단위 메타 동봉).
worklog/011 § D.5 의 ExtractedImage 정규화 항목에 추가 필요.

### 4-4. samples/*.gt.json 의 expected_image_count 갱신

samples/ 는 `.gitignore` 라 git 영향 없지만, **사용자 로컬의 gt 파일이
outdated** 한 경우 회귀 테스트 실패. 이번 patch 에서 3개 갱신:
- `samples/pptx_letsportal_tmp.pptx.gt.json` → 0 → 46
- `samples/pptx_qvan_storyboard.pptx.gt.json` → 0 → 499
- `samples/●제안서(최종)_한국관광공사_*.pptx.gt.json` → 0 → 38

각 gt 에 `_v0.4.4_note` 필드로 변경 이력 박음. 새 PC 에서 samples 받으면
같은 갱신 필요 — **scripts/refresh_gt.py 자동화는 v0.5+ 후보**.
