## 2. 설계 결정

### 2-1. worklog/007 § 2-3 의 "미진입" 진단 재검토

| 표면 (v0.2 시점) | 진짜 (v0.4.5 시점 재검토) |
|---|---|
| "HWP 의 페이지 좌표계와 ExtractedImage.bbox 의 PDF-style 좌표계가 비자명 매핑" → 미진입 | **bbox 매핑이 문제였을 뿐, bytes/sha256 추출은 가능.** worklog/016 § 2-4 의 PPTX 결정 (EMU 단위 그대로, 통일은 v0.5 Pipeline) 과 같은 논리. HWP 는 좌표 매핑 자체가 없으므로 `bbox=None` 으로 정직하게 박음 |

→ v0.2 미진입은 **합리적이었음** (bbox 정책이 미정), v0.4.5 진입은 **PPTX 의
결정 (단위 불일치 OK, Pipeline 정규화 미루기) 이 같은 카테고리 문제를 해소**
했기 때문에 가능.

### 2-2. bindata/ 디렉토리 → ExtractedImage 매핑

pyhwp 의 `HTMLTransform.transform_hwp5_to_dir` 가 BinData stream 을
`{tmpdir}/bindata/BIN{NNNN}.{ext}` 패턴으로 export. ext 는 PNG / JPG / BMP
(samples 5종 실측).

| ExtractedImage 필드 | 값 |
|---|---|
| `page_no` | **None** (XHTML 안 이미지 anchor → 페이지 매핑 불가) |
| `section_no` | None |
| `bbox` | **None** (HWP 좌표계 → PDF-style bbox 매핑 없음) |
| `order_in_page` | bindata 파일명 sort 순서 (1-based). pyhwp 의 stream 번호 보존 |
| `text_before` / `text_after` / `section_title` | "" / "" / None |
| `file_path` | tempfile (`kdp_hwp_{order:04d}_*.{ext}`) — bindata 가 곧 삭제되므로 별도 영구 복사 |
| `sha256` | blob sha256 (64-char hex) |
| `width / height` | PIL Image 사이즈. 실패 시 (0, 0) |
| `size_bytes` | `len(blob)` |
| `mime_type` | `mimetypes.guess_type(suffix)` — `image/png` / `image/jpeg` / `image/bmp` |
| `detected_caption` / `caption_method` / `caption_pattern_score` | None / None / 0.0 |

### 2-3. tempfile 영속화 — bindata 가 곧 삭제됨

`tempfile.TemporaryDirectory()` 가 `with` 블록 종료 시 자동 삭제 →
**bindata 의 파일들을 그 시점에 별도 tempfile 로 복사해야 함**.
`_collect_bindata_images(bindata_dir)` 호출은 with 블록 안에서 수행, 반환은
영구 tempfile path 리스트.

→ `parse()` 의 `with tempfile.TemporaryDirectory() as tmpdir:` 안에서
`html = html_path.read_text(...)` 직후, `images = _collect_bindata_images(...)`
호출. with 종료 후 markdown / tables 가공.

### 2-4. graceful degrade — bindata/ 디렉토리 없음

pyhwp 의 일부 케이스 / 이미지 0건 HWP / 손상 케이스에서 `bindata/` 가 아예
없을 수 있음. `_collect_bindata_images(missing_dir)` 가 빈 list 반환 — 텍스트
파싱은 그대로 성공. 단위 테스트 (`test_hwp_extract_returns_empty_bindata_gracefully`).

### 2-5. SemVer — patch (0.4.4 → 0.4.5)

PPTX (v0.4.4) 와 같은 결정: `ParseResult.images` 의 HWP 동작 변경 (`[]` → N건)
은 호환 가능한 채움. v0.5 의 minor 적정성을 위해 자산화 patch 들은 모두
v0.4.x patch 로 유지 (worklog/014 § 4-2, worklog/016 § 2-6 의 패턴 계승).

### 2-6. `_safe_image_size()` 코드 중복 — 의도적

PPTX 와 HWP parser 가 같은 8줄 helper 를 각자 가짐. 이유:
- packages/hwp 가 packages/core/parsers 의 module-private 함수를 import 하는
  건 패키지 경계 침범 (CLAUDE.md §"패키지 경계 §5")
- 공유 helper 를 core 에 두려면 새 모듈 (`parsers/_imageutil.py`) 필요 — v0.4.5
  범위 늘림
- 8줄짜리 utility 의 의도적 중복은 단일 파일 포터빌리티 원칙과도 정합

→ **v0.5+ refactor 후보** (worklog/016 § 2-4 와 같은 시점에 처리).

## 4. 잠재 과제 / 다음 세션이 알아야 할 것

### 4-1. HWP 이미지의 page_no / bbox 정보 복원 가능성

현재 HWP 의 이미지는 페이지/위치 정보 0건. **이론적 복원 경로:**
- pyhwp 의 `Hwp5File` 의 record-level 접근 (`bodytext/section*.xml`) →
  paragraph 별 `<picture>` element + binary ref → 어느 페이지에 어떤 BinData 가
  anchor 되어 있는지 매핑 가능
- 다만 페이지 개념이 HWP 의 layout-driven 페이지네이션 (정확한 페이지 번호는
  렌더 시점에 결정) 이라 정확한 page_no 보존이 비자명
- bidwatch 의 사용처가 페이지 정보를 요구하면 v0.5+ 진입 후 결정

### 4-2. 같은 이미지의 중복 (HWP 의 dedup 정책)

`bindata/BIN0001.png` 가 HWP 본문에 여러 번 anchor 되어 있어도 BinData stream
은 1개 → v0.4.5 의 추출도 1개. PPTX 의 슬라이드 중복 (worklog/016 § 4-1) 과
다른 양상 — **HWP 가 BinData 측면에서 native dedup 되어 있음**.

→ 운영상 이점: 같은 이미지가 100번 등장하는 HWP 도 Vision 비용 1회만.

### 4-3. `.bmp` 의 BMP 처리 — Vision API 호환성

samples 5종에서 `image/bmp` 다수. Claude Vision API 가 BMP 를 직접 지원
하는지 확인 필요:
- worklog/011 B.2 가 "비트맵 ≥ 100x100" 박았지만 mime 정책은 명시 안 함
- 만약 Anthropic 이 BMP 거부하면 `kdp-label` 호출 시 에러 — graceful
  처리 필요 (skip + warn, v0.4.6 또는 v0.5 patch)

→ **bidwatch 운영 진입 전 확인 필요**. 단순 검증: `kdp-label image.bmp` 한
번 호출 (단위 비용 ~1원). v0.4.6 의 첫 작업 후보.

### 4-4. `_safe_image_size()` refactor — v0.5+

PPTX / HWP / (잠재적으로 추가 포맷) 의 8줄 helper 중복. v0.5 진입 시
`parsers/_imageutil.py` 또는 `core.py` 의 helper 로 통합. 단일 진실원.

### 4-5. samples/*.hwp.gt.json 갱신

samples/ 는 `.gitignore` 라 git 영향 없지만, 사용자 로컬의 gt 파일이
outdated 한 경우 회귀 테스트 실패. 이번 patch 에서 5개 갱신:
- `samples/test_nara.hwp.gt.json` → 0 → 12
- `samples/proposal_gyeongnam_fishery.hwp.gt.json` → 0 → 18
- `samples/3. 정성적 제안서_*.hwp.gt.json` → 0 → 22
- `samples/proposal_wku.hwp.gt.json` → 0 → 55
- `samples/proposal_forest_startup.hwp.gt.json` → 0 → 91

각 gt 에 `_v0.4.5_note` 필드로 변경 이력 박음. worklog/016 § 4-4 의 PPTX gt
갱신과 같은 패턴. **scripts/refresh_gt.py 자동화는 v0.5+ 후보**.
