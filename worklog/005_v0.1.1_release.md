## 3. v0.1.0 의 자기-비판 — "합성 어렵고 실 한/글 파일 의존성 회피"

`worklog/002_v0.1.0_release.md` 의 Known limitations 항목 중:

> **HWPX 이미지 픽스처 미포함** — 합성 어렵고 실 한/글 파일 의존성 회피.

이번에 실제로 합성 작업해보니 **1시간 작업**이었음. ZIP 에 PNG/JPEG 바이트를
`BinData/` 또는 `Contents/Resources/` 경로로 넣기만 하면 됨. v0.1.0 시점에는
"합성 어려움" 으로 적었지만 진짜 근거는 **시간 부족 + 우선순위 밀림** 이었음.

**표면 근거 ≠ 진짜 근거** 사례가 또 발견됨 (HANDOVER §4-4 PPTX 정책 흔들림과
동형). 다음 마일스톤에서 "X 가 어렵다" 라고 적게 될 때 30분 정도 실제로 시도해본
뒤 어렵다고 결론짓는 절차가 필요. 안 그러면 사실 빠르게 해결될 일이
"미해결" 상태로 누적됨.

## 5. 잠재 과제 / 다음 세션이 알아야 할 것

### 5-1. PDF 비트맵 추출 — v0.2 첫 의사결정 후보

3-way 비교 필요 (의존성 추가 결정이므로 CLAUDE.md 의무):

| 후보 | 라이선스 | 의존성 무게 | 구현 난도 | 운영 위험 |
|---|---|---|---|---|
| pypdf | BSD-3 | 가벼움 (~2MB) | 낮음 | 없음 |
| pdfplumber 내부 stream 직접 | 추가 0 | 0 | **높음** (ColorSpace/Filter 분기) | PNG/JPEG 외 깨짐 |
| pdf2image (poppler) | MIT + binary | **무거움** | 낮음 | 외부 binary 의존, CI 무거움 |

v0.2 시작 시 LibreOffice 의존성과 함께 패키지 매트릭스를 한 번에 결정.

### 5-2. HWPX 미커버 4개 라인 (96% → 100% 추구 여부)

미커버: `hwpx.py` 76, 115→95, 144→141, 146.
- 76: `ParseError` 재-raise (정상 ParseError 가 발생하는 path 픽스처 미보유)
- 115→95, 144→141: 빈 paragraph / 빈 stripped 분기 (edge case)
- 146: title `<opf:title>` 의 `text` 가 빈 문자열인 케이스

**의도적으로 100% 추구 안 함.** 95% 게이트 통과로 충분. 새 분기가 추가될 때 자연
도달. 100% 강박은 **테스트가 코드를 모방하는 안티패턴** 의 입구라서 회피.

### 5-3. mypy "unused section" 경고

`pyproject.toml` 의 mypy override 4개 (`markitdown`, `pyhwp`, `pywintypes`,
`win32com`) 가 현재 `unused section(s)` 경고. v0.2 에서 해당 라이브러리 import
시작되면 자연 해소. 지금 지우면 v0.2 에서 다시 추가해야 함 → **유지**.

### 5-4. v0.2 진입 전 확인 사항

- bidwatch 일정 (이번 세션에서 사용자 답변 미수령). 일정에 따라 v0.2 우선순위
  (HWP 먼저 vs LibreOffice 4포맷 먼저) 가 갈림.
- kordoc 차용 검토 (`worklog/004`) — HWP 작업 시 pyhwp 한계 드러나면 진입.
