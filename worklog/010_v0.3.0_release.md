## 3. markitdown 의존성의 진짜 비용 분석

worklog/009 의 §3-1 (운영자 비용 비대칭) 가 markitdown 에도 적용. 하지만 격리됨:

| 부담 | 영향 범위 |
|---|---|
| **80MB ML 스택** (onnxruntime + numpy + magika) | **PPTX 안 쓰는 사용자: 0**. PPTX 사용자: 1회 install |
| Docker 이미지 layer | PPTX 사용 서비스만 +80MB (캐싱) |
| CI 시간 | PPTX 픽스처 7건 추가 → 전체 회귀 ~2분 증가 (1분 59초 → 4분 16초) |
| 런타임 메모리 | markitdown 호출 중 numpy 일시 로드 (~50MB) |
| 안정성 | magika file-type detection 이 PPTX 라우팅 결정 — corrupt bytes 의 합성 테스트는 우회되어 vanasso 실 케이스로 대체 |

→ **PPTX extras 의 진짜 부담은 "PPTX 를 쓰는 서비스만 짊어진다"**. worklog/009 의 boundary 1종 원칙 (Python 패키지만) 도 유지됨 (system binary 없음).

## 4. 표면 근거 vs 진짜 근거

### 표면 근거
- Phase 0 벤치마크: markitdown 23K vs 자체 170 → 위임 자명
- 도메인 신호: bidwatch 4위 + 사내 RAG 3위 → PPTX 필요

### 진짜 근거
1. **인터페이스 일관성 보존** — markitdown 의 슬라이드 마커 + markdown 표 형식이 RAG 청킹 / Vision 라벨링 anchor 로 활용 가능. 자체 구현으로는 슬라이드 단위 분할 어렵고 표 추출 사실상 불가능
2. **유지보수 비용 절감** — PPTX 의 OOXML 스펙은 복잡 (마스터 슬라이드 / 차트 / SmartArt / Notes). markitdown 위임 시 그 복잡도를 Microsoft 가 유지보수. 우리는 어댑터 ~100줄만 관리
3. **2단계 평가로 위험 통제** — 평가 단계가 마련되어 있어 "도입 후 발견되는 문제" 대신 "도입 전 사전 검증" 패턴. v0.4 의 Pipeline 작업 시 동일 패턴 재사용 가능
4. **80MB 가 격리됨** — extras 옵트인이라 코어 사용자 영향 0. HWP[hwp] 와 동일 패턴 (사용자가 install 시 명시적 동의)

## 5. PPTX 가 가져온 새로운 contract

v0.3.0 의 `ParseResult` for PPTX:
- `markdown`: markitdown 의 슬라이드별 출력 (`<!-- Slide number: N -->` 마커 + markdown 표 + 이미지 placeholder)
- `tables`: **empty** (v0.3 contract — markdown 본문에 인라인. 향후 ParsedTable 승격은 v0.4+)
- `images`: **empty** (v0.3 contract — markdown 본문에 `![](file.png)` placeholder. 비트맵 추출은 v0.4 의 Vision 작업과 동기)
- `metadata.title`: python-pptx core_properties 기반 best-effort

이는 PDF/DOCX/HWPX/HWP 의 contract 와 다름 — 후자는 `tables` / `images` 가 구조화된 list 로 반환됨. PPTX 는 markitdown 위임의 자연스러운 결과로 다른 contract.

다운스트림(bidwatch / 사내 RAG) 은 PPTX 처리 시 `markdown` 만 사용하는 것을 권장. 표 데이터가 구조화되어 필요하면 v0.4 의 ParsedTable 승격을 기다리거나, markdown 표를 후처리하여 list 로 변환.

## 6. 다음 세션이 알아야 할 것 (v0.4 진입 가이드)

### 6-1. v0.4 의 첫 의사결정 — Pipeline 패키지 시작

`packages/pipeline/` 의 skeleton 만 있는 상태. v0.4 작업:
1. `DocumentIngestionPipeline` 클래스 — extract() 결과를 받아 상태 머신 (pending → auto_approved / needs_review / ...) 으로 관리
2. Claude Vision 라벨링 — `ExtractedImage` 의 caption / 분류
3. 검수 큐 API (FastAPI) — 사용자가 검수할 항목 목록
4. 검수 UI (HTML/JS) — 단축키 + 일괄처리

v0.3 의 2단계 평가 패턴을 v0.4 에도 적용. Claude API 비용이 발생하므로 평가 단계에서 **샘플 N≥5 비용 사전 측정** 의무.

### 6-2. PPTX 의 후속 과제

- **표 → ParsedTable 승격** — markitdown 의 markdown 표를 다시 파싱하여 list of list 로 복원. 단점: markdown 표는 cell 안 줄바꿈 / merged cell 처리 불가. v0.4 의 Vision 라벨링과 트레이드오프 평가 필요
- **이미지 placeholder → ExtractedImage** — markitdown 은 이미지 placeholder 만 emits. 비트맵 자체는 python-pptx 로 직접 추출 필요. Vision 라벨링과 동기 작업 (placeholder 와 비트맵 매핑)
- **Notes 슬라이드** — markitdown 이 `### Notes:` 형식으로 추출. 다운스트림이 본문/노트 구분하길 원하면 별도 필드로 분리 검토

### 6-3. HWP 회귀 시간 (worklog/008 의 미해결 과제)

v0.3.0 회귀 시간: 4분 16초 (v0.2.1 의 1분 59초 + PPTX 7건 추가 ~2분). pytest-xdist
병렬화 검토는 v0.4 patch 후보로 보류 — Windows multiprocessing 의 fixture
직렬화 복잡도 검토 필요.

### 6-4. lets_portal 마이그레이션 (Phase 2)

bidwatch 미정이라 lets_portal 마이그도 미정. v0.4 의 Pipeline 출시 후 bidwatch
가 실 사용 결정하면 그때 Phase 2 진입.
