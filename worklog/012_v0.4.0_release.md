## 3. mock 테스트 53건의 가치

실 API 호출 0건으로 코드 검증:

- `_FakeAnthropic` 클래스가 SDK 의 `messages.create` 응답을 scripted reply 로 대체
- 53건 테스트가 다음 분기 모두 커버:
  - 1-pass 성공 (confidence ≥ threshold)
  - 2-pass 성공 (confidence < threshold → reasoning 추가)
  - 캐시 hit (2번째 호출이 API 우회)
  - PII 마스킹 (010-1234-5678 → 010-XXXX-XXXX)
  - JSON 파싱 실패 → fallback 값
  - markdown fence (```json ... ```) 자동 strip
  - 캐시 미사용 모드 (--no-cache)
  - CLI 인자 파싱 + missing file + 단일 이미지 mode

→ **vision 모듈 99% coverage 가 실 API 비용 0원으로 달성됨**.

worklog/011 § 9 의 합격 기준 (N≥5 평가) 은 ANTHROPIC_API_KEY 가 필요해 v0.4.0

### 평가 실행 가이드

```bash
# 1. API 키 설정
$ export ANTHROPIC_API_KEY=sk-ant-...

# 2. 단일 이미지 시험
$ kdp-label samples/test_image.png

# 3. 실 PPTX/PDF 에서 추출된 이미지 일괄 라벨링
$ kdp-label --from-document samples/proposal.pdf --output eval.jsonl

# 4. 비용 / 한국어 비율 / confidence 분포 측정 (수동)
```

### 합격 기준 재게재 (worklog/011 § 9)

| 지표 | 합격선 |
|---|---|
| caption 한국어 비율 | hangul ≥ 30% / caption |
| image_type 정확도 | 정답 일치 ≥ 4/5 (사용자 수동 검증) |
| confidence 분포 | auto_approved (≥0.7) 비율 60-80% |
| 이미지당 비용 | ≤ $0.01 (Sonnet) |
| Haiku 비교 | hangul / 정확도 둘 다 < Sonnet 의 80% 인지 |

5건 모두 합격 → v0.5 Pipeline 진입. 1건이라도 크게 실패 → 모델 / 라벨링 범위
재논의.

## 5. 잠재 과제 / 다음 세션이 알아야 할 것

### 5-1. 실 평가 1회 후 임계값 조정 가능성
- B.1 의 0.7 default 가 너무 높으면 → 60% auto_approved 미달, 검수 부담 ↑
- 0.7 이 너무 낮으면 → 검수 가치 ↓ (모든 게 자동 승인)
- 평가 결과로 0.65 ~ 0.85 사이에서 조정

### 5-2. Haiku 비교 측정 권고
- 5건 같은 이미지 × Sonnet + Haiku 동시 호출 = 10건 비용 ($0.05 + $0.005)
- Haiku 가 합격선 통과하면 비용 1/10 절감 가능 — v0.4.1 patch 후보

### 5-3. CLI 의 미커버 56% line coverage
- `_run_from_document` (문서 모드) 가 실 extract() 호출 + Vision 호출이라 mock 부족
- 실 평가 시점에 자연 커버됨
- 의도적 — TestPyramid 의 mock vs e2e 트레이드오프

### 5-4. v0.5 진입 전 결정 사항
- PostgreSQL 운영 환경 (사내 / Docker / 클라우드)
- 비동기 worker 기술 (asyncio / Celery / RQ)
- v0.4 SQLite 캐시의 데이터 마이그 전략 (rows → PostgreSQL)
- ExtractedImage 에 ai_caption / ai_image_type 등 v0.4 결과 통합 (현재는 별도)
