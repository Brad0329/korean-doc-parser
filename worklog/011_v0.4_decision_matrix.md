## 2. 카테고리 A — 비용 모델

### A.1 Claude 모델 = **Sonnet**

#### 표면 근거
- 한국어 caption 품질 + 비용 균형

#### 진짜 근거
- **홈페이지 자산화 시나리오**: 초기 spike (대량 1회) + long tail (정기 갱신)
  - 초기 1만 이미지 = $100 (Sonnet) — 회사 자산 가치 대비 무시할 수준
  - long tail 월 200장 = $2 — 무시 가능
- **bidwatch = 유료 모델**: end user 가 비용 부담 → 사용료에 Vision 비용 반영 가능
  - 1문서 (20 이미지) = $0.20 — 충분히 가격 반영 가능
- Haiku 는 한국 차트 caption 품질 우려, Opus 는 사치
- v0.4 Vision CLI 평가에서 **Haiku 도 동시 측정** 후 retreat 가능

#### 추가 가성비 옵션 (v0.4 평가 단계 적용)
1. **Prompt Caching** (Anthropic 자동) — 같은 system prompt 반복 시 cached 토큰 비용 90% 절감. 대량 처리 시 효과 큼
2. **Message Batches API** — 24시간 안 처리, **50% 할인**. 홈페이지 초기 자산화에 적합 (비실시간)
3. **Haiku 동시 측정** — 같은 5건 이미지로 Sonnet vs Haiku 품질 비교

### A.2 비용 상한 = **코드 안 없음 + Anthropic Console spending limit (외부 backup)**

#### 진짜 근거
- **홈페이지 자산화** = 사용자 본인이 trigger → 사고 시나리오 1 (무한 루프 업로드) 자동 차단
- **bidwatch** = 상위 레이어 유료 결제 게이트 → 사고 시나리오 2 (악의적 대량 사용) 자동 차단
- **sha256 캐시 (C.4)** → 사고 시나리오 4 (캐시 미스 폭주) 자동 차단
- 남은 시나리오 3 (코드 버그) 만 외부 console 의 spending limit 으로 backup

#### 외부 console 활용
- console.anthropic.com → Settings → Spend Limit
- 월 $X 도달 시 API 자동 차단
- 코드 안 상한 장치 불필요

### A.3 상한 초과 동작 = **N/A**
A.2 가 "코드 안 상한 없음" 이라 자동 N/A.

### A.4 비용 추적 = **메모리 객체 + Pipeline DB cost_log + 호출자 DB 옵션**

#### 진짜 근거 (사용자 통찰)
- **라이브러리는 데이터 객체로 반환만, 저장은 사용 측 책임** 이 원칙
- Pipeline 패키지는 **예외적으로 DB** — 여러 호출 간 상태 (검수 큐 / 비용 집계) 필요
- `packages/core/hwp/pptx` 는 stateless, `packages/pipeline` 만 stateful

#### 3계층 추적
1. **메모리 객체**: `ExtractedImage.ai_cost_krw` 필드 (라이브러리 반환값)
2. **Pipeline DB**: `cost_log` 테이블 (`timestamp / image_sha256 / model / usage / cost_krw / doc_id`)
3. **호출자 DB**: bidwatch / 사내 RAG 가 옵션으로 통합 (`ai_cost_krw` 가져가서 자기 invoice 에 반영)

#### v0.4 의 단순화
- v0.4 = Vision CLI 만 → DB 불필요, **stdout JSON 출력만**
- v0.5 = Pipeline 진입 시 PostgreSQL cost_log 구축

## 3. 카테고리 B — 라벨링 범위

### B.1 라벨링 필드 = **caption + image_type + confidence (1호출, 모든 이미지) + reasoning (신뢰도 < 임계값 시 2차 호출)**

#### 표면 근거
- 동일 비용으로 4필드 채울 수 있다면 모두 채움 (사용자 직관)

#### 진짜 근거 + 비용 분석
- 1호출 4필드 vs 4호출 분리: **1호출이 2배 효율** (이미지 4번 보낼 필요 없음)
- reasoning 추가 비용 = +30-50% (절대값 $0.003/이미지) — 작지만 대량 시 누적
- reasoning 의 진짜 가치: **needs_review 만 필요** (auto_approved 는 안 봄)

#### 2-pass 방법 (default)
```
1차 호출: caption + type + confidence (reasoning 없음)
  ↓ confidence >= 0.7 → 끝 (auto_approved 후보)
  ↓ confidence < 0.7  → 2차 호출 (같은 이미지, reasoning 추가)
```
- 평균 비용 절감 ~20% (E.1 임계값 0.7 가정 시 70% auto_approved 비율)

#### v0.4 의 평가 사항
- 2-pass vs single-call conditional output (모델 instruction 으로 처리) — 어느 것이 정확/싼가

### B.2 라벨링 대상 = **비트맵 이미지만, 임계값 100x100 px**

#### 표면 근거
- 작은 아이콘/로고 제외 (RAG 가치 없음)

#### 진짜 근거
- **시장 표준과 동급** — markitdown / unstructured / docling 모두 비트맵만
- 도형/차트/SmartArt = 페이지 PNG → Vision 필요 (LlamaParse / Mistral OCR)
  - 페이지 PNG 변환 = **모든 길이 system binary 필요** (LibreOffice / GhostScript / Poppler) → **worklog/009 영구 금지**
  - 유일한 Python-only 옵션 = PyMuPDF (AGPL, ~40MB) — v0.5+ 의 별도 결정
- 시간 부담: 페이지 단위 라벨링 시 분 단위, **비동기 필수** (D.2 와 정합)

#### 향후 결정
- v0.5+ 에서 도형/차트 진짜 필요 판명 시 PyMuPDF [pdf-render] extras 추가 검토

### B.3 언어 = **한국어 only**

#### 진짜 근거
- 비용 절감 (병기 대비 -15%, 절대값 작음)
- 단순함 + 한국 도메인 우선
- 향후 영문 사용자 필요 판명 시 patch 검토 (영구 결정 아님)

### B.4 OCR = **미진입**

#### 진짜 근거
- **의존성 부담** — Tesseract = worklog/009 금지 system binary, PyTorch OCR = 수백MB
- boundary 1종 원칙 (Python 패키지 only) 유지
- caption 만으로 RAG 검색 충분
- 향후 필요 시 Claude Vision OCR 통합으로 가벼운 추가 가능

## 4. 카테고리 C — 데이터 보안

### C.1 외부 전송 동의 = **전체**

#### 진짜 근거 (사용자 통찰)
- **홈페이지 자산화** = 사용자 본인이 수행 → trigger 시점에 본인 동의 자명
- **bidwatch** = end user 가 업로드하는 거라 사용 자체에 송신 동의 포함
- 도메인별 / 사용자 동의 체크박스 등은 UI 차원, 라이브러리 책임 밖

### C.2 PII 처리 = **caption 결과 post-process 마스킹**

#### 충돌 발견 → 재결정
- 원래 "자동 마스킹" 결정은 OCR 필요 → **B.4 OCR 미진입과 충돌**
- 이미지 안 PII 검출 = OCR 없이 불가능

#### 재결정 — 옵션 2
- Vision 응답의 caption 텍스트에 regex 검사 (주민번호 / 전화 / 이메일 패턴)
- 검출 시 마스킹 (`010-XXXX-XXXX`)
- 이미지 자체는 외부 송신됨 (Claude 가 본 것 = 결과만 보호)
- 효과 제한적이지만 운영 부담 작음

### C.3 API 키 = **환경변수 (ANTHROPIC_API_KEY)**

#### 진짜 근거
- Anthropic SDK 표준 — 코드 안 추가 설정 불필요, 자동 읽음
- Docker / k8s secret / .env 등 모든 배포 환경과 호환
- 사내 secret store 도입은 향후 옵션 (현 시점 최소 의존성)

### C.4 캐싱 = **sha256 + model 조합**

#### 진짜 근거 (사용자 통찰)
1. **sha256 필드 이미 존재** (v0.1.0 부터) — 추가 구현 부담 거의 0
2. **검수자 의견 일치 = 운영 함문과 일치** — 동일 이미지 두 번 결과가 다르면 검수자 혼란
3. 캐시 = **사고 시나리오 4 (캐시 미스 폭주) 자동 차단** + **D.4 롤백 시 비용 보호**

#### 캐시 정책 디테일
- Key: `sha256 + model` (Sonnet → 다음 모델 업그레이드 시 자동 새 캐시)
- 위치: PostgreSQL `image_label_cache` 테이블 (v0.5+), v0.4 는 SQLite 임시
- 만료: 영구 (sha256 동일 = 결과 동일 가정)

## 5. 카테고리 D — Pipeline 인터페이스

### D.1 저장소 = **PostgreSQL** (Pipeline 패키지 한정)

#### 진짜 근거 (사용자 통찰 + 메인 분석)
Pipeline 의 6가지 DB 필요 use case:
1. **검수 큐 상태 머신** (pending/processing/auto_approved/needs_review/...) — 여러 호출 간 상태 → **필수**
2. **비용 추적** (cost_log) — 집계 / 감사
3. **sha256 캐시** (C.4) — 영구 보존, 재시작 후에도 비용 보호 → **필수**
4. **검수 이력** (E.3 수집)
5. **사용자 / 권한** (E.4 — 1인이라 옵션)
6. **재시도 / 실패 로그** (retry queue / dead letter)

→ **Pipeline 자체 PostgreSQL 거의 필수**. 호출자 DB 통합은 진짜 옵션.

#### v0.4 의 단순화
- v0.4 = Vision CLI → DB 불필요 (SQLite 임시 캐시만)
- v0.5 = Pipeline 진입 시 PostgreSQL 본격 도입

### D.2 인터페이스 = **비동기**

#### 진짜 근거
- **web 서비스 HTTP 패턴 표준** — submit → job_id 반환 → polling/webhook
- Vision 라벨링 = 분 단위 → 동기로는 end user 응답 타임아웃
- Anthropic API rate limit 관리 / 재시도 / 장애 복구 자연스러움

#### v0.4 의 단순화
- v0.4 CLI = 동기 (stdout 즉시 출력)
- v0.5 Pipeline = 비동기 (background worker)

### D.3 멱등성 = **doc_id = sha256(file bytes) 자동 채택**

#### 진짜 근거
- **중복 업로드 차단** — 사용자 실수로 다시 올려도 비용 한 번만
- **네트워크 재시도 안전** — API 실패 후 재요청 시 자동 대처
- **호출자 부담 0** — 라이브러리가 sha256 자동 계산

### D.4 트랜잭션 = **부분 실패 시 롤백**

#### 진짜 근거
- **재시도 멱등성** — 증분적 재평가 (어디부터?) 복잡도 회피
- **DB 상태 일관성** — "완료/실패" 명확
- **C.4 캐시가 비용 보호** — 롤백되어도 sha256 캐시는 별도 보존 → 재시도 시 50% 절감

## 6. 카테고리 E — 검수 큐 정책

### E.1 자동 승인 = **신뢰도 ≥ 0.7**
- Claude 의 confidence 활용
- ~70% auto_approved + ~30% needs_review (가정)
- v0.4 Vision CLI 평가에서 confidence 분포 측정 후 임계값 조정

### E.2 정렬 = **신뢰도 낮은 순**
- needs_review 큐 안에서 가장 의심스러운 것 우선
- 모델 약점 빠른 발견, 검수 가치 극대화

### E.3 재학습 = **미수행 (수집만)**
- 수정 이력 DB 저장 (E.1 의 needs_review → user_approved 전환 시 변경 내역)
- 데이터 누적 후 v0.5+ 에서 few-shot 또는 fine-tune 결정

### E.4 권한 = **1인 (v0.4-v0.6)**
- 로그인 / 권한 시스템 불필요
- 다인 / 워크플로우는 v0.7+ 에서 사용처 (bidwatch 멀티 운영자) 확정 시
