## 2. 표면 근거 vs 진짜 근거

### 2-1. 표면 근거

- 파서 라이브러리 안에 PostgreSQL / 비동기 큐 / 검수 UI 가 들어있는 게 어색
- 사내 private 라이브러리 (CLAUDE.md 정체성) 와 어울리지 않음

### 2-2. 진짜 근거

| 차원 | 진단 |
|---|---|
| **자체 원칙 위반** | worklog/011 § A.4 가 이미 "라이브러리는 데이터 객체 반환만, 저장은 사용 측 책임" 박음. Pipeline 의 "예외적으로 DB" 는 이 원칙의 자기 모순 |
| **HANDOVER § 2-1 위반** | "하층 (Engine): DB/AI/UI 의존 0건". Pipeline 이 상층이라 해도 같은 라이브러리 안. 호출자 관점에서 라이브러리 import 시 PostgreSQL 의존 발생 |
| **호출자 입장 비효율** | bidwatch / vanasso.kr / 사내 RAG 가 어차피 자체 DB 보유. Pipeline 의 PostgreSQL 까지 띄우면 **두 개 DB 관리** — 운영 비용 + 동기화 문제 |
| **검수 UI 의 결합도** | UI 는 호출자의 인증 / 권한 / 디자인과 강결합. 라이브러리가 UI 제공하면 호출자가 따라야 하는 강제. bidwatch 의 어드민 ≠ 사내 RAG 의 어드민 |
| **v0.5 작업량** | (a 안 = Pipeline 패키지 삭제) / (b 안 = 알고리즘만) / (c 안 = 원안 유지) 중 c 안의 80% 가 PostgreSQL/큐/UI. b 안은 작업량 1/3 로 축소 |
| **사내 1인 + bidwatch 메인 사용처** | 사용처가 명확히 알려진 상황에서 "범용 Pipeline" 만들 정당성 약함. bidwatch 본인이 본인 schema 로 박는 게 정직 |

→ 표면 근거가 약화돼도 (예: 회사가 커져서 여러 사용처가 동일 DB schema 원하게 됨)
진짜 근거 1~3 은 그대로 유지. **결정의 내구성** 확보.

## 3. v0.5.0 의 산출물 정의 (b 안 확정)

### 3-1. 새 모듈 위치

```
packages/core/src/korean_doc_parser/pipeline/
├── __init__.py          # public re-exports
├── caption.py           # caption 1차 검출 (regex + proximity)
├── confidence.py        # 3중 가중 신뢰도 (pure function)
└── doc_id.py            # compute_doc_id(path) → sha256(file)
```

**왜 `packages/core/.../pipeline/` (core 안 sub-package)?**
- 별도 패키지 (`packages/pipeline/`) 는 worklog/011 § 8 의 마일스톤 split 원안
- b 안에서는 stateless 알고리즘 모듈만이므로 **core 의 sub-module** 로 충분
- `packages/pipeline/` 디렉토리 신설은 over-engineering (PostgreSQL/큐 없으면
  packages 경계 분리 정당성 약함)
- 단, sub-package 이름은 `pipeline` 유지 — 호출자 코드에서 의미 명확

### 3-2. 각 모듈 contract

#### `caption.py`

```python
def detect_caption_regex(text: str) -> tuple[str | None, float]:
    """주변 텍스트에서 '그림 1', 'Figure 2', '<표 1>' 등 패턴 매칭.
    반환: (caption, pattern_score 0.0-1.0)"""

def detect_caption_proximity(
    image_bbox: tuple[float, float, float, float] | None,
    text_blocks: list[TextBlock],
    bbox_unit: str,  # "px" | "emu" | "none"
) -> tuple[str | None, float]:
    """bbox 거리 기반 가장 가까운 텍스트 블록 = caption 후보."""
```

→ 각 파서 (PDF / PPTX / HWP / HWPX / DOCX) 가 자신의 ExtractedImage 채울 때
호출. `detected_caption` / `caption_method` / `caption_pattern_score` 필드 채움.

#### `confidence.py`

```python
def weighted_confidence(
    *,
    regex_score: float,         # 0.0-1.0
    proximity_score: float,     # 0.0-1.0
    vision_confidence: float,   # 0.0-1.0
    weights: tuple[float, float, float] = (0.4, 0.2, 0.4),
) -> float:
    """3중 가중 합산 (worklog/011 § B). Pure function, DB 0건."""
```

→ 호출자가 받은 `ExtractedImage.caption_pattern_score` + bbox proximity 점수 +
`VisionResult.confidence` 를 합성. 호출자의 어떤 threshold 로 분기할지는
호출자 자유 (worklog/014 의 임계값 0.7 권장).

#### `doc_id.py`

```python
def compute_doc_id(path: Path | str) -> str:
    """sha256(file_bytes). 호출자의 primary key 후보. Pure function."""
```

## 5. 호출자 (bidwatch / vanasso.kr) 입장에서 무엇이 달라지는가

### 5-1. 변하지 않는 것 (v0.4.5 까지의 사용 패턴 유지)

- `extract(path)` → `ParseResult` (markdown / tables / images / metadata)
- `VisionClient(model=..., cache=...)` → `VisionResult`
- `kdp-label` CLI 동작
- SQLite cache 의 위치 / 형식

### 5-2. v0.5.0 에서 새로 가능한 것

- `from korean_doc_parser.pipeline import compute_doc_id, weighted_confidence`
  로 알고리즘 helper 호출
- 각 파서가 caption 1차 검출까지 자동 수행 (이전엔 `detected_caption=None`
  였던 필드가 채워짐)
- `kdp-label` 의 default 모델이 Haiku — **비용 1/10, 사용자 코드 변경 0건**
  (이전 명시적으로 `--model claude-sonnet-4-5` 안 썼다면 자동 적용)
- `ExtractedImage.bbox_unit` 으로 단위 메타 확인 가능
- `kdp-label --stats` 가 cache hit_rate 도 표시

### 5-3. v0.5.0 에서 사라지는 것 (원래 worklog/011 에 박혀 있다가 폐기된 항목)

호출자 입장에서 **"기대했는데 안 나오는 것"** 0건. 원래 박혀 있던 PostgreSQL
/ 큐 / 검수 UI 는 모두 호출자가 자체 구현해야 할 항목이므로, **라이브러리가
제공 안 한다는 사실이 더 명확해짐** 이 호출자에게 이득.

## 6. 잠재 과제 / 다음 세션이 알아야 할 것

### 6-1. proximity score 계산의 bbox_unit 의존

`detect_caption_proximity()` 가 bbox 거리를 계산하려면 단위가 통일돼야 함.
v0.5.0 의 `bbox_unit` 필드가 enabler:
- PDF (`bbox_unit="px"`) → 픽셀 거리
- PPTX (`bbox_unit="emu"`) → EMU 거리 (1 inch = 914400 EMU)
- HWP / HWPX (`bbox_unit="none"`) → proximity 계산 불가, score=0.0

**v1.0+ 후보:** HWP 의 record-level 접근으로 bbox 복원 (worklog/017 § 4-1).
지금은 HWP 의 proximity = 0.0 으로 두고, 3중 가중에서 vision 비중이 커짐.

### 6-2. caption 1차 검출의 한국어 패턴

regex 패턴 후보:
- `<그림 \d+>` / `\[그림 \d+\]` / `그림 \d+`. (HWP / DOCX 빈도 높음)
- `<표 \d+>` / `\[표 \d+\]`
- `Figure \d+` / `Fig. \d+` / `Table \d+` (영문 혼용)
- `자료: ...` / `출처: ...` (PDF 빈번)

samples/ 5종 도메인에서 실측 후 패턴 finalize. **v0.5.0 작업 안 의사결정 항목**.

### 6-3. weights = (0.4, 0.2, 0.4) 의 정당성

worklog/011 § B 가 박은 가중치. 그러나 실 운영 데이터 누적 후 재조정 필요.
**v0.5.0 의 weights 는 default 일 뿐, 호출자 customizable** 로 박음
(weighted_confidence 의 `weights` 인자).

### 6-4. benchmarks/adapters/ 의 PPTX 미반영 (worklog/018 § 4-1)

v0.4.5 벤치마크에서 PPTX 4 fixture skip. v0.5.0 안 또는 v1.0 전 처리.
adapter 코드 갱신 + v0.5.0 baseline 재측정 권고.

### 6-5. archive 정리의 worklog 분류 기준

Phase 7 의 archive 작업에서 worklog/ 17건을 분류해야 함. 기준:
- **archive 로 이동:** 처리된 마일스톤 release 기록 (000, 002, 005, 007, 008,
  010, 012)
- **archive 로 이동 + 잠재 과제 추출:** 의사결정 worklog 의 § "잠재 과제"
  부분만 `docs/internal/known-limitations.md` 로 발췌 (011, 014, 015, 016,
  017, 018, 019 등)
- **유지:** 가장 최신 결정 + 살아있는 잠재 과제 reference

Phase 7 진입 시 명세화.
