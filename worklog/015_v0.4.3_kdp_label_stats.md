## 2. 설계 결정

### 2-1. 기존 모드와 mutex — argparse 그룹이 아니라 main() 분기

`--stats` 가 first 분기. image / `--from-document` 와 동시 입력 시 stats 가 이김.
argparse `add_mutually_exclusive_group` 는 positional (nargs="?") 과 호환 안 됨.
→ main() 안의 `if args.stats: return _run_stats(args)` 로 단순화.

### 2-2. cache 파일이 없으면 silent create 거부

`VisionCache(path)` 가 자동으로 빈 DB + schema 를 만들어버림. `--stats` 가
typo 한 경로에 빈 DB 를 silently 만들면 사용자가 혼란. → `_run_stats()` 에서
`Path.is_file()` 체크 후 친절한 에러 메시지 + exit 1.

### 2-3. cache hit rate 측정은 v0.4.3 범위 밖

worklog/013 § 5-2 권고는 "캐시 적중률 / 일별 비용". 적중률은 cache 의 row
schema 에 `hit_count` 컬럼이 없어서 측정 불가. schema 변경은 v0.4.3 범위 초과
+ 일정 시간 데이터 누적도 필요. → **v0.5.0 의 PostgreSQL 마이그레이션에서
schema 재설계 시 같이**. v0.4.3 은 "누적 상태 스냅샷" 만.

### 2-4. saved_krw 의 정의

`saved_krw` = "cache 에 박힌 row 의 누적 cost_krw" = "캐시가 없었다면 다시
지불했을 비용". 사용자가 같은 이미지를 재요청하면 그만큼 절약된 셈. cost 가
아니라 saved 로 명명한 이유는 캐시 hit 시 0원으로 처리되는 운영 직관에 맞춤.

### 2-5. by_date 정렬

SQLite `GROUP BY date(created_at)` + `ORDER BY d`. Python 의 dict 는 insert
order 보존 → JSON 출력도 오래된 날짜부터. 운영자가 최근 추세 보기 쉬움.

## 3. 무엇이 측정되는가

| 필드 | 의미 |
|---|---|
| `db_path` | cache 파일 경로 (typo 디버깅용) |
| `total_rows` | unique (sha256, model) 의 누적 개수 |
| `total_saved_krw` | 모든 row 의 cost_krw 합 |
| `by_model` | 모델별 {rows, saved_krw}. Sonnet vs Haiku 분포 확인 |
| `by_date` | YYYY-MM-DD 별 {rows, cost_krw, by_model}. 사용 spike 감지 |
| `last_7_days_saved_krw` | 최근 7일 총 saved (inclusive of today) |

---

## 4. 잠재 과제 / 다음 세션이 알아야 할 것

### 4-1. v0.5.0 의 PostgreSQL schema 에 반영할 항목

- `hit_count` 컬럼 — cache hit 시 +1, 진짜 적중률 측정 가능
- `last_accessed_at` — 콜드 row 정리 (TTL 정책) 근거 데이터
- `caller` 컬럼 — bidwatch / 사내 RAG 등 호출자 식별 (worklog/011 D.4 의 cost_log
  와 연계)

### 4-2. CLI 인체공학 — 표 형식 출력

JSON 만 지원. 운영자가 사람 친화 표를 원하면 `jq` 로 처리하거나, v0.5+ 에서
`--format table` 추가. v0.4.3 범위 최소화.

### 4-3. 자산화 경로의 다음 단계

worklog/014 § 5-1 (PPTX 이미지 추출) 또는 worklog/007 § 2-3 (HWP 이미지
추출). 자산화 우선 경로에서 v0.4.4 또는 v0.4.5 후보.
