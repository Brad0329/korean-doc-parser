## 3. 표면 결과 vs 진짜 분석

### 표면

- 회귀 0건, 21% 개선 → v0.5 진입 OK

### 진짜

#### 3-1. HWP 이미지 추출 추가에도 더 빠른 이유

worklog/017 의 `_collect_bindata_images()` 가 pyhwp 의 `bindata/` 디렉토리
walk + tempfile 복사 + sha256 + PIL size 만 수행. **무거운 변환 0건** (그냥
파일 복사 + 해시). pyhwp 의 XHTML transform 시간이 전체의 80%+ 이고, 이건
v0.2 부터 동일.

#### 3-2. v0.2 대비 21% 개선의 진짜 원인 추정

| 가능성 | 진단 |
|---|---|
| v0.2 측정 시점의 백그라운드 부하 | v0.2 측정은 1회뿐이라 noise 추적 어려움 |
| 의존성 (pypdf / hwp5 / bs4) 자체 최적화 | v0.4.x 동안 의존성 업그레이드 있을 수 있음 — uv.lock diff 로 확인 가능하지만 본 worklog 범위 밖 |
| 같은 PC, 다른 시점의 시스템 부하 | 가장 유력 — Windows 의 file cache / antivirus 등 |

→ "v0.4.5 가 진짜로 21% 빠르다" 보다는 **"성능 회귀 없음, 노이즈 영역 변동"**
으로 해석. **회귀 0건 = ±5% 게이트 통과** 가 핵심.

#### 3-3. PDF 의 composite score 0.5 / 0.75 — v0.2 부터 동일

`스테이블코인 법제화.pdf` 의 composite = 0.500 (text_len=0 으로 ground truth
매칭 실패). 키움증권 0.750. 둘 다 **v0.2 부터 동일** — v0.4.x 회귀 아님.
이 PDF 들은 스캔 PDF 라 텍스트 추출 한계가 있음 (v0.5+ OCR 통합 후보).

## 4. 부수 발견 / 잠재 과제

### 4-1. fixtures: 13 vs runs: 9 — adapter 가 PPTX 미반영

`compare.py` 의 fixture walker 가 `.pptx` 도 detect (13개) 하지만, korean_doc_parser
adapter (`benchmarks/adapters/korean_doc_parser.py` 추정) 가 **v0.3 의 PPTX 지원
반영 안 함** → 4 PPTX fixture skip. corrupt PPTX 1건도 skip 됐을 가능성.

**영향:** PPTX/HWP 이미지 추출의 회귀 측정이 누락. 다음 측정 시점부터 adapter
업데이트 필요.

→ **부채로 박힘 (v1.0 전 처리 또는 v0.5 안에서 같이).**

### 4-2. `legacy_file_parser` adapter 가 ok 인데 사용 안 됨

v0.4.5 adapter 자체 점검에서 `legacy_file_parser` = available. 이건 lets_portal
의 기존 file_parser.py 와의 회귀 비교용. 본 측정은 `--adapters korean_doc_parser`
로 한정해서 비교 의도와 정합 — 단, **v1.0 전 마지막 측정 시 legacy 와의
회귀 비교** 권고 (lets_portal 마이그레이션 영향).

### 4-3. ±5% 게이트의 적용 단위 재검토

현 게이트는 "mean_ms ±5%" 가 가장 직관적이지만 노이즈에 약함 (3-2 참고).
**v0.5 안에서 게이트 정의 정밀화 후보:**
- 같은 PC, 같은 부하 조건 보장 (CI runner 표준화)
- N 회 반복 측정 → 중앙값
- fixture 당 ±5% 가 아니라 **수렴 기반** (예: ±2 sigma)

본 worklog 범위 밖. v1.0 stable lock 직전에 결정.
